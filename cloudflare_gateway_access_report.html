3. تحليل القدرة على الوصول

3.1 ما يمكن الوصول إليه فعلياً

عناصر يمكن الوصول إليها بدون مصادقة

التالي متاح للعامة ويمكن فحصه بدون أي بيانات مصادقة:

  • Team Domain App Launcher: yugalabs.cloudflareaccess.com - صفحة تطبيقات الفريق
  • JWKS Endpoint: يعيد المفاتيح العامة لتوقيع JWT
  • OpenID Configuration: يكشف عن قدرات OIDC للفريق
  • صفحات إعادة التوجيه: 302 redirect إلى صفحة تسجيل الدخول
  • CF_AppSession Cookies: كوكيات جلسة مؤقتة (تتغير مع كل طلب)
  • JWT Headers: معلومات رأس JWT غير مشفرة

3.2 ما لا يمكن الوصول إليه بدون مصادقة

عناصر محمية تتطلب مصادقة صالحة

التالي لا يمكن الوصول إليه بدون بيانات مصادقة صالحة (Okta SSO، بريد إلكتروني مع OTP، إلخ):

  • staging.otherside.xyz - بيئة الاختبار (محتوى حقيقي قد يحتوي بيانات حساسة)
  • sandbox.otherside.xyz - بيئة التجريب
  • api.sandbox.otherside.xyz - واجهة برمجة التطبيقات التجريبية
  • JWT Payload - الحمولة الكاملة لرمز المصادقة (تحتوي على هوية المستخدم)
  • موارد التطبيقات الداخلية - أي بيانات أو خدمات خلف البوابة

5.2 التحليل التفصيلي لأخطر نواقل الهجوم

ناقل #1: استغلال مسار delegatedToken (الخطر الأكبر)

مستوى الخطر: حرج | صعوبة التنفيذ: متوسطة

تم تفكيك الكود المصدري الفعلي من layout-9e5e982d526c9efb.js (Module 7809) واكتشاف أن مسار ?delegatedToken=XXX يمرر التوكن عبر Server Action يسمى exchangeDelegatedTokenForPrivyTokens. هذا المسار يعمل خارج حماية Cloudflare Access - لأنه جزء من الموقع العام (otherside.xyz) وليس ضمن التطبيقات المحمية.

إذا تمكن مهاجم من اعتراض delegatedToken (عبر Referer header أو XSS أو سجل المتصفح)، يمكنه استخدامه للحصول على accessToken + refreshToken + identityToken من Privy، وبالتالي الوصول إلى محفظة المستخدم الرقمية بشكل كامل دون الحاجة للمرور عبر بوابة Cloudflare Access.

ناقل #2: اختراق حساب Okta SSO

التقارير تكشف عن 6 حسابات Okta مؤكدة لفريق Yuga Labs، بما في ذلك حساب admin@yuga.com بصلاحيات إدارية. إذا تم اختراق أي حساب Okta عبر:

  • تصيد احتيالي (Phishing) مستهدف
  • هجوم SIM Swap لالتفاف 2FA
  • استغلال ثغرات Okta نفسها
  • سرقة بيانات الاعتماد من تسريبات أخرى

فإن المهاجم يحصل على وصول كامل عبر Cloudflare Access إلى جميع التطبيقات المحمية، لأن Cloudflare Access يستخدم Okta كمزود هوية (IdP) رئيسي.

ناقل #3: CORS Misconfiguration على نقاط غير محمية

نقطتا api.otherside.xyz وassets.otherside.xyz ليستا محميتين بـ Cloudflare Access وتسمحان بـ CORS: *. رغم أنهما تعيدان AccessDenied حالياً، فإن إعدادات CORS المفتوحة تعني أن أي موقع ويب يمكنه إرسال طلبات إليهما. إذا تمت إضافة ملفات جديدة إلى حاوية S3 الخلفية، ستكون قابلة للوصول من أي مصدر.

7. الاستنتاجات والتوصيات

7.1 الاستنتاج الرئيسي

هل يمكن الوصول إلى Cloudflare Gateway؟

الإجابة المختصرة: نعم، يمكن الوصول إلى البوابة، لكن لا يمكن تجاوزها.

بوابة Cloudflare Access تعمل كحارس وصول (Zero Trust Gateway) بشكل صحيح وفعال. يمكن الوصول إلى واجهة البوابة (Team Domain، JWKS، OpenID Config) بدون مصادقة حسب التصميم، لكن جميع التطبيقات المحمية تتطلب مصادقة صالحة عبر Okta SSO أو طريقة مصادقة أخرى معتمدة. البوابة تعيد توجيه المستخدمين غير المصادقين بشكل صحيح عبر 302 redirect وتُنشئ كوكيات جلسة آمنة (Secure + HttpOnly).

7.2 نقاط الضعف المكتشفة (خارج البوابة نفسها)

رغم أن البوابة نفسها تعمل بشكل صحيح، تم اكتشاف عدة نقاط ضعف في النظام البيئي المحيط التي يمكن أن تُستغل للوصول غير المباشر:

7.3 التوصيات العاجلة

  1. إزالة delegatedToken من URL: استبدال ?delegatedToken= بـ POST body أو cookie مشفرة - هذا الخطر الأكبر لأنه يعمل خارج حماية Cloudflare Access
  2. إضافة Referrer-Policy: strict-origin-when-cross-origin لمنع تسريب URL parameters عبر Referer
  3. إضافة CSP شامل: Content-Security-Policy يمنع تنفيذ سكريبتات غير مصرح بها وسرقة التوكنات عبر XSS
  4. إضافة SRI لسكريبت Plausible.io: منع اختراق سلسلة التوريد
  5. إصلاح HSTS: إضافة includeSubDomains و preload
  6. تقييد CORS: على api.otherside.xyz و assets.otherside.xyz
  7. تفعيل Token One-Time-Use: كل delegatedToken يُستخدم مرة واحدة فقط
  8. إضافة Token Binding: ربط delegatedToken ببصمة الجلسة (IP + User-Agent)
الخلاصة النهائية

بوابة Cloudflare Access نفسها آمنة وتعمل بشكل صحيح كحارس وصول Zero Trust. المشكلة الحقيقية ليست في البوابة، بل في مسارات المصادقة البديلة (delegatedToken و customAuth) التي تعمل خارج نطاق حماية البوابة. هذه المسارات تسمح للمستخدمين بتجاوز Cloudflare Access بالكامل للحصول على توكنات Privy المصادقة. يجب إغلاق هذه المسارات البديلة أو حمايتها بنفس مستوى حماية البوابة.
