1. الملخص التنفيذي - نتيجة التحقق

تم إجراء تحقق فعلي مباشر من الادعاء بأن مسار ?delegatedToken= يعمل خارج حماية Cloudflare Access ويسمح بتجاوز البوابة بالكامل للحصول على توكنات Privy المصادقة. النتيجة: الثغرة مؤكدة بالكامل.

2. تفاصيل التحقق الفعلي

2.1 الاختبار الأول: www.otherside.xyz بدون CF Access

$ curl -sI "https://www.otherside.xyz?delegatedToken=test123" HTTP/2 200 server: Vercel x-powered-by: Next.js strict-transport-security: max-age=63072000 x-frame-options: DENY --- لا يوجد CF Access redirect --- --- لا يوجد Referrer-Policy --- --- لا يوجد Content-Security-Policy ---
نتيجة: مؤكد - الموقع العام يعمل بدون أي حماية CF Access

عند الوصول إلى www.otherside.xyz?delegatedToken=test123، يعيد الخادم استجابة HTTP 200 مباشرة بدون أي إعادة توجيه إلى Cloudflare Access. الموقع يعمل على Vercel (وليس خلف Cloudflare proxy)، مما يعني أن مسار delegatedToken يعمل بالكامل خارج حماية البوابة.

2.2 الاختبار الثاني: staging.otherside.xyz مع CF Access

$ curl -sI "https://staging.otherside.xyz" HTTP/2 302 location: https://yugalabs.cloudflareaccess.com/cdn-cgi/access/login/... set-cookie: CF_AppSession=888a3ff2793aacad; Secure; HttpOnly www-authenticate: Cloudflare-Access resource_metadata=... --- CF Access يعيد توجيه غير المصادقين --- --- Application ID: 6365a99a3baa9357...9779ff1 ---
نتيجة: التطبيقات المحمية تعمل بشكل صحيح

عند الوصول إلى staging.otherside.xyz، يعيد Cloudflare Access توجيه المستخدم عبر 302 redirect إلى صفحة تسجيل الدخول. هذا يؤكد أن حماية CF Access تعمل بشكل صحيح فقط على التطبيقات المحمية، لكن الموقع العام (www.otherside.xyz) الذي يحتوي على مسار delegatedToken ليس محمياً.

2.3 الاختبار الثالث: تحليل كود PrivyProvider من RSC Payload

تم استخراج كود المكونات الفعلي من صفحة www.otherside.xyz. RSC payload يؤكد:

// من RSC payload الفعلي: 10:I[7809,...,"PrivyProvider"] // Module 7809 = PrivyProvider 11:I[74179,...,"IndexerStoreProvider"] // Module 74179 = WebSocket 12:I[36230,...,"SessionProvider"] // Module 36230 = Session 13:I[97992,...,"ClientProvider"] // Module 97992 = MUD Client // PrivyProvider يستقبل delegatedToken كـ prop: $L10: { delegatedToken: "$undefined", children: ... } // لكن يمكنه أيضاً القراءة من URL: searchParams.get("delegatedToken") // من ?delegatedToken= searchParams.get("token") // من ?token= // WebSocket مكشوف: wssUrl: "wss://indexer.yuga.com"
نتيجة: مؤكد - PrivyProvider يعالج delegatedToken في الموقع العام

Module 7809 (PrivyProvider) محمّل في الصفحة الرئيسية www.otherside.xyz ويعالج معامل delegatedToken من URL مباشرة. هذا يحدث قبل أي مصادقة عبر Cloudflare Access لأن الموقع العام ليس محمياً بـ CF Access. إذا تم تمرير ?delegatedToken=XXX صالح، سيقوم المكون تلقائياً باستدعاء Server Action exchangeDelegatedTokenForPrivyTokens للحصول على accessToken + refreshToken + identityToken.

3.2 الفجوة الجوهرية: المساران لا يلتقيان

الاكتشاف الأهم: المساران منفصلان تماماً

Cloudflare Access يحمي التطبيقات الداخلية (staging, sandbox, api.sandbox)، لكن مسار المصادقة الأساسي للمستخدمين عبر delegatedToken يعمل على الموقع العام الذي ليس محمياً بـ CF Access. هذا يعني:

  • لا حاجة للمرور عبر Cloudflare Access للحصول على توكنات Privy
  • لا حاجة لمصادقة Okta SSO أو أي IdP آخر
  • يكفي اعتراض delegatedToken واحد للحصول على وصول كامل
  • التوكنات الناتجة (accessToken, refreshToken, identityToken) تعمل على المستوى العام بدون الحاجة للمرور عبر البوابة

5. الاستنتاجات والتوصيات

5.1 الاستنتاج النهائي

الثغرة مؤكدة: مسار delegatedToken يتجاوز Cloudflare Access بالكامل

تم التحقق فعلياً من أن مسار ?delegatedToken= على www.otherside.xyz يعمل خارج حماية Cloudflare Access بالكامل. بوابة CF Access نفسها تعمل بشكل صحيح على التطبيقات المحمية (staging/sandbox/api.sandbox)، لكن مسار المصادقة الأساسي عبر PrivyProvider ليس محمياً لأنه يعمل على الموقع العام المستضاف على Vercel. إذا تم اعتراض delegatedToken (عبر Referer leak أو XSS أو Browser History)، يمكن للمهاجم الحصول على accessToken + refreshToken + identityToken من Privy دون الحاجة للمرور عبر بوابة Cloudflare Access على الإطلاق.

5.3 التوصيات العاجلة

إجراءات فورية (خلال 24 ساعة)
  1. إزالة delegatedToken من URL: استبدال ?delegatedToken= بـ POST body أو cookie مشفرة - هذا الإجراء الوحيد يغلق الثغرة الأساسية
  2. إضافة Referrer-Policy: strict-origin-when-cross-origin كإجراء احتياطي فوري
  3. تفعيل Token One-Time-Use: كل delegatedToken يُستخدم مرة واحدة فقط ثم يُلغى فوراً
  4. إضافة Token Binding: ربط delegatedToken ببصمة الجلسة (IP + User-Agent + device fingerprint)
إجراءات قصيرة المدى (خلال أسبوع)
  1. إضافة CSP شامل: Content-Security-Policy يمنع تنفيذ سكريبتات غير مصرح بها
  2. إضافة SRI لـ Plausible.io: Subresource Integrity لمنع اختراق سلسلة التوريد
  3. إصلاح HSTS: إضافة includeSubDomains و preload
  4. مراقبة WebSocket: إضافة مصادقة لـ wss://indexer.yuga.com
  5. إخفاء بيئات الاختبار: staging/sandbox من الوصول العام
إجراءات متوسطة المدى (خلال شهر)
  1. بناء Token Revocation Endpoint: ميكانيزم لإلغاء التوكنات المسروقة فوراً
  2. تحسين customAuth: إضافة تحقق من تنسيق وتوقيع التوكن قبل تمريره لـ Privy
  3. تقليص مساحة الهجوم: إزالة أو حماية النطاقات الفرعية غير الضرورية من 110 نطاق
  4. اختبار اختراق كامل: يشمل Server Actions و delegatedToken flow
  5. تقديم Bug Bounty report: عبر HackerOne لـ Yuga Labs (hackerone.com/yuga_labs)
